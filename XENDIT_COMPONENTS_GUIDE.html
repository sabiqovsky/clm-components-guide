2. How the Demo Store Works (End to End)

A step-by-step journey of a customer named Ayu buying a plushie:

  1. Ayu browses the store — Picks the "Joyful Badger" ($20), clicks "Add to Cart".
  2. Ayu clicks "Checkout" — The store sends cart details (items, quantity, currency) to the merchant's server.
  3. The server talks to Xendit — Calls the Xendit Sessions API to create a payment session:
    • Payment Link → Returns a URL; Ayu is redirected to Xendit's checkout page.
    • Components → Returns a components_sdk_key; payment form appears on the merchant's own page.
    • Invoice (Legacy) → Returns a URL; similar to Payment Link using the older API.
  4. Ayu pays — Enters card details (on the merchant's page for Components, on Xendit's page for Payment Link).
  5. Success — Ayu sees "Payment Successful".

    Store->>Ayu: Shows "Payment Successful!" page

3. What Are Xendit Components?

Xendit Components lets merchants accept payments directly on their own website. Instead of redirecting the customer to a Xendit-hosted page, a secure payment form is embedded right inside the merchant's checkout page.

Why do merchants want this?

Benefit Explanation
Brand consistency Payment form lives on the merchant's site — looks and feels like theirs
Higher conversion No redirect means fewer drop-offs
Full layout control Merchant decides where and how the form appears
Still secure Card details handled securely by Xendit, not the merchant

How is it different from Payment Link?

Aspect Payment Link Xendit Components
Where customer pays Xendit's hosted page (redirected) Merchant's own website
Look and feel Xendit's branding Merchant's branding
Setup effort Minimal — just redirect Moderate — needs front-end embedding
Customisation Limited Full — merchant controls styling and layout

Key events the front-end listens for

These are status signals from Xendit that tell the page what to show:

Event What it means What the page does
init Components is ready Show the card input form
submission-ready All required fields filled Enable the "Pay Now" button
submission-not-ready Fields missing or invalid Disable the "Pay Now" button
submission-begin Payment processing Show a loading spinner
submission-end Processing finished Hide the spinner
session-complete Payment succeeded Show "Success!" page
fatal-error Critical failure Show an error message

Event lifecycle diagram

stateDiagram-v2
    [*] --> Loading: Page loads
    Loading --> Ready: init event
    Ready --> FormNotReady: submission-not-ready
    Ready --> FormReady: submission-ready
    FormNotReady --> FormReady: submission-ready

    Submitting --> Processing: submission-begin
    Processing --> FormReady: submission-end (failed, can retry)
    Processing --> Success: session-complete
    Loading --> Error: fatal-error
    Processing --> Error: fatal-error
    Success --> [*]

Important details for merchants

  1. API key stays on the server. It is a secret — never sent to the browser.
  2. components_sdk_key goes to the front-end. Temporary, session-specific key that only allows displaying the payment form.
  3. Origins must be declared. The session specifies which website URL can use Components (e.g., https://mystore.com).
  4. Multi-currency support. Each currency (IDR, PHP, MYR, etc.) uses its own API key via environment variables.

6. How Xendit Components Can Be Styled

Merchants can style Components to match their brand. There are two layers of customisation:

Layer 1: The container (fully customisable)

The wrapper around the payment form. Merchant has complete control over:

  • Background colour, border, border radius
  • Padding, spacing, width, positioning
  • Shadow effects

Layer 2: The input fields (customisable via appearance settings)

Card input fields live inside a secure Xendit iframe, but the merchant can still customise passing variables in the SDK configuration:

What can be customised Examples
Text colour Match brand colour (e.g., #252525)
Font family Same font as the site (e.g., Inter, Roboto)
Placeholder text colour Subtle hints (e.g., #7d7d7d)
Custom web fonts Load Google Fonts or any external web font

Demo store example

  • Text colour: #252525 (dark charcoal)
  • Font: Inter
  • Placeholder colour: #7d7d7d
  • Font loading: Fetch from Google Fonts

7. Security & PCI-DSS Compliance

What is PCI-DSS?

Payment Card Industry Data Security Standard — strict security rules for businesses handling credit card data. Non-compliance can result in:

  • Heavy fines from card networks (Visa, Mastercard, etc.)
  • Loss of ability to accept card payments
  • Reputational damage from data breaches

How Xendit Components help

The merchant's website never touches card data. Card details are entered inside a secure Xendit-hosted iframe and sent directly to Xendit — bypassing the merchant's server entirely.

Key talking points for merchant conversations

  1. "Do I need PCI-DSS compliance?" — Using Components drastically reduces PCI burden. Merchants benefit from Xendit's PCI-DSS Level 1 certification.
  2. "Is it safe?" — Yes. Card data is captured in a secure iframe and sent directly to Xendit over encrypted connections.
  3. "What about the components_sdk_key?" — Temporary, session-specific. Cannot be used to make payments or access card data.
  4. "What about the API key?" — Stays on the server. Never exposed to the browser.

8. Merchant FAQs

General
Q: Does the customer get redirected when using Components?

No. The customer stays on the merchant's website. The payment form is embedded in the page.

Q: Can I use Components for all payment methods?

Components currently support cards. The SDK lists available channels; merchants display forms for active channels on their account.

Q: What currencies are supported?

Depends on the merchant's Xendit account. The demo supports IDR, MYR, PHP, SGD, THB, VND.

Q: What payment flows are supported?
  • Pay — One-time payment
  • Save — Save a payment method (no charge)
  • Pay & Save — Pay and save the method
Integration & Setup
Q: How hard is it to integrate?

Moderate effort — more than Payment Link, but gives full control over checkout. Requires:

  1. A server calling the Xendit Sessions API
  2. A front-end loading xendit-components-web to render the form
Q: Does the merchant need to install anything?

The dev team installs the xendit-components-web npm package. No plugins or extensions needed.

Q: Does it only work with React?

No. Works with any front-end framework or plain HTML/JavaScript. The SDK returns standard HTML elements.

Q: Is there a sandbox/test mode?

Yes. Merchants use test API keys for simulated transactions. The live demo itself is a test environment.

Styling & Customisation
Q: Can the merchant change the look of the form?

Yes, at two levels:

  1. Container — Full CSS control (colours, borders, spacing, size)
  2. Input fields — Text colour, font, placeholder colour, custom web fonts
Q: Can the merchant move or resize the form?

Yes. The form is an HTML element inside a container. The merchant controls size and position with CSS.

Q: Can the merchant change the button?

Yes. The "Pay Now" button is not part of Xendit Components — the merchant creates their own button and calls submit().

Security
Q: Does the merchant see or store card numbers?

No. Card data is entered inside a secure Xendit-hosted iframe. The merchant's website and server never access it.

Q: Do merchants still need PCI-DSS compliance?

Components significantly reduce PCI scope. Merchants should still follow basic security practices (HTTPS, software updates).

Q: What happens on a payment error?

The SDK fires an error event. The customer can retry without starting over.

Q: Is the components_sdk_key sensitive?

Treat it as sensitive. It's temporary and session-specific — don't log, save, or share it.

Troubleshooting
Q: Components form isn't showing up?

Common causes:

  1. Wrong origin — URL must match the origins declared when creating the session
  2. Expired SDK key — Sessions have a limited lifespan
  3. Network issues — Browser must reach Xendit's servers
Q: "Pay Now" button is greyed out?

The customer hasn't filled all required fields. Button activates when all fields are valid.

Q: Payment failed after clicking "Pay Now"?

Possible reasons: card declined, network error, or 3D Secure cancelled. The customer can retry with the same or a different card.

Q: Can the customer retry after a failure?

Yes. The form returns to its previous state — no need to start over.
